# Incident Investigation – SSH Brute Force Attack

## 📌 Executive Summary

This investigation documents a simulated SSH brute-force attack against the Linux server (`MyDFIR-Linux-Tee`) within the MyDFIR SOC environment.

The objective was to validate the complete SOC workflow by:

- Detecting repeated authentication failures
- Triggering automated alerts
- Performing alert triage
- Investigating attacker activity
- Determining whether the alert represented a true security incident

The investigation was performed using Elastic Security, Kibana Discover, Sysmon telemetry, Linux authentication logs, and custom detection rules.

# Detection Summary

## Detection Rule

```text
MyDFIR-SSH-Brute-Force-Attempt-Tee
```

Purpose:

Detect multiple failed SSH authentication attempts against the Linux server.

---

## Detection Logic

The rule monitors failed SSH authentication events generated from the Linux authentication logs.

### Detection Query

```kql
system.auth.ssh.event : * and
agent.name : "MYDFIR-Linux-Tee" and
system.auth.ssh.event : "Failed" and
user.name : "root"
```

# Alert Validation

The investigation began by opening the alert generated by Elastic Security.

The alert indicated repeated failed SSH authentication attempts targeting the **root** account.

Initial validation confirmed:

- Detection rule triggered correctly
- Multiple failed login attempts occurred
- Linux endpoint was reporting telemetry successfully

---

# Investigation Methodology

## Step 1 — Review Authentication Events

Opened Kibana Discover and filtered authentication logs using the detection query.

Verified:

- Failed authentication events
- Username targeted
- Event timestamps
- Source IP address

---

## Step 2 — Identify Target Account

Reviewed:

```text
user.name
```

Observed:

```text
root
```

This confirmed the attack targeted the Linux administrative account.

---

## Step 3 — Review Source Information

Reviewed:

- source.ip
- source.geo.country_name

Objective:

Determine:

- Attack origin
- Geographic location
- Repeat activity from the same IP

---

## Step 4 — Review Authentication Pattern

Examined:

```text
system.auth.ssh.event
```

Confirmed:

- Repeated "Failed" authentication events
- No successful authentication observed during the investigation

This reduced the likelihood of a successful compromise.

---

## Step 5 — Correlate Related Events

Reviewed:

- Linux authentication logs
- Kibana Discover
- Authentication dashboard
- Network map visualization

Confirmed all telemetry aligned with the generated alert.

# True Positive Analysis

The alert was classified as a **True Positive** because:

- Multiple failed SSH login attempts were observed.
- Authentication events matched the detection rule.
- The activity originated from an external source.
- The targeted account was the privileged **root** account.
- The activity was consistent with a brute-force attack simulation.

---

# False Positive Considerations

Potential false positives could include:

- An administrator repeatedly entering an incorrect password.
- Automated infrastructure health checks.
- Configuration testing by another authorized administrator.

In this lab environment, the activity correlated directly with the planned attack simulation and was therefore confirmed as malicious.

---

# Findings

The investigation identified:

- Multiple failed SSH authentication attempts.
- Repeated targeting of the **root** account.
- No successful authentication.
- Detection rule functioning as expected.
- Dashboard visualizations accurately representing attack activity.

---

# Analyst Assessment

The attack represented an attempted credential attack but did **not** result in unauthorized access.

The Linux host remained uncompromised during this investigation.

The detection rule successfully identified suspicious authentication activity before a successful login occurred.

---

# Lessons Learned

This investigation reinforced several SOC investigation practices:

- Authentication failures should always be correlated with successful login events.
- Source IP reputation and geographic location provide useful investigative context.
- Monitoring privileged accounts such as **root** increases detection effectiveness.
- Threshold-based detections reduce noise while highlighting repeated attack patterns.
